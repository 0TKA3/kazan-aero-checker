# aero

Веб-приложение и CLI для просмотра онлайн-расписания аэропорта Казани ([kazan.aero](https://www.kazan.aero/on-line-schedule/)).

## Требования

- [Bun](https://bun.com) v1.3+
- **Firefox** — только как fallback, если cookie-авторизация API перестанет работать (Selenium)

## Установка

```bash
bun install
cd web && bun install
```

## Веб-приложение

Запуск API-сервера и фронтенда одной командой:

```bash
bun run dev
```

- UI: [http://localhost:5173](http://localhost:5173)
- API: [http://localhost:3000](http://localhost:3000)

Отдельный запуск:

```bash
bun run dev:server   # API на :3000
bun run dev:web      # React UI на :5173
```

Сборка фронтенда:

```bash
bun run build
```

Продакшн-запуск (API + собранный UI на одном порту):

```bash
bun run build
bun run start:prod
```

Откроется на [http://localhost:3000](http://localhost:3000).

## Docker

Запуск на сервере через Docker Compose:

```bash
docker compose up -d --build
```

Приложение будет доступно на порту **3000** (или задайте свой через переменную `PORT` в `.env`).

```bash
# другой порт, например 8080
PORT=8080 docker compose up -d --build
```

Проверка:

```bash
curl http://localhost:3000/api/health
```

Остановка:

```bash
docker compose down
```

Только Docker без Compose:

```bash
docker build -t aero .
docker run -d --name aero -p 3000:3000 --restart unless-stopped aero
```

### Возможности UI

- Календарь для выбора даты
- Вкладки «Вылет» / «Прилёт»
- Поиск по номеру рейса, авиакомпании, направлению
- Таблица рейсов с цветными статусами
- Детальная карточка: регистрация, стойки, выходы, посадка
- Светлая и тёмная тема
- Кнопка «Обновить» — сброс кэша и повторный запрос

## REST API

| Метод | Путь | Описание |
|-------|------|----------|
| GET | `/api/health` | Статус сервера и источника данных |
| GET | `/api/flights?date=DD.MM.YYYY` | Список рейсов на дату |
| GET | `/api/flights?date=...&type=departure` | Только вылетающие |
| GET | `/api/flights?date=...&type=arrival` | Только прилетающие |
| GET | `/api/flights?date=...&q=N4` | Поиск |
| GET | `/api/flights?date=...&refresh=true` | Принудительное обновление кэша |
| GET | `/api/flights/:flightNumber?date=...` | Детали одного рейса |

Пример:

```bash
curl "http://localhost:3000/api/flights?date=17.06.2026&type=departure"
```

## CLI

Программа принимает два аргумента:

1. **Номер рейса** — поле `flight_number_iata` из API (например, `N4 705`, `SU 1234`)
2. **Дата** — поле `date` из API, в формате `DD.MM.YYYY` (например, `19.06.2026`)

```bash
bun run cli "N4 705" 19.06.2026
# или
bun run index.ts "N4 705" 19.06.2026
```

Кавычки вокруг номера рейса обязательны, если в нём есть пробел.

### Формат аргументов

Поиск идёт по **точному совпадению** с данными API — программа не нормализует и не угадывает формат.

| Поле | Правильно | Неправильно |
|---|---|---|
| Номер рейса | `"N4 705"` | `N4705`, `N4-705`, `705` |
| Дата | `19.06.2026` | `2026-06-19`, `19/06/2026` |

В API у каждого рейса есть несколько полей с номером:

- `flight_number_iata` — **используется программой** (`N4 705`, `SU 1234`)
- `flight_number_icao` — ICAO-код (`NWS 705`, `AFL 1234`)
- `flight_number` — локальное название (`КЛ 705`, `СУ 1234`)

### Частые ошибки

**Склеенный номер рейса.** Nordwind N4 705 часто вводят как `N4705`. В API это два отдельных значения — код авиакомпании `N4` и номер `705`, разделённые пробелом.

**Неверный формат даты.** API возвращает дату как `19.06.2026`, а не `2026-06-19`. Даже если рейс существует, с неправильной датой поиск не сработает.

**Пропущены кавычки.** Без кавычек shell передаст `N4` и `705` как два отдельных аргумента:

```bash
# Неправильно — shell разобьёт аргументы
bun run index.ts N4 705 19.06.2026

# Правильно
bun run index.ts "N4 705" 19.06.2026
```

### Как узнать правильные значения

Открой [расписание на kazan.aero](https://www.kazan.aero/on-line-schedule/) или посмотри ответ API. Найди нужный рейс и скопируй значения полей `flight_number_iata` и `date` без изменений.

При успешном поиске в консоль выводится таблица с данными о рейсе. Если рейс не найден — `Рейс не найден`.

## Архитектура

```
aero/
├── src/
│   ├── lib/          # загрузка, кэш, фильтрация
│   ├── server/       # Bun HTTP API
│   └── cli/          # CLI-обёртка
├── web/              # React + Vite + shadcn/ui
└── index.ts          # entrypoint CLI
```

Данные загружаются с `https://www.kazan.aero/json/dld.json` через cookie-авторизацию и кэшируются на 5 минут. Selenium/Firefox используется только как fallback.

## Примечания

- Номер рейса и дата должны **точно совпадать** с полями `flight_number_iata` и `date` в API.
- Если аргументы CLI не переданы, программа завершится без вывода.
- Вкладки «Вылет» / «Прилёт» определяются по статусу рейса (`Прибыл` → прилёт, остальное → вылет).
